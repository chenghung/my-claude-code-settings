## 定位

此 skill 提供分層的文件研究流程。Main agent 在 skill 中先評估研究範圍與複雜度，決定需要啟動多少個一手研究 subagent，以及是否需要二手整合 subagent 做跨來源彙整。實際的搜尋與整合工作由對應 subagent 全權執行，main agent 只負責拆解、調度與最終決策，不自行執行搜尋或整合。

## 觸發時機

應觸發的情境：

- 使用者明確要求查找技術文件或 library 用法
- 需要 API spec 或外部 library 的具體行為規格
- 需要研究某技術主題的最新做法或最佳實踐
- Main agent 在任務執行中途發現需要外部資訊驗證才能繼續推進

不應觸發的情境：

- 僅憑現有 context 即可回答的問題，不需外部查找
- 純粹的程式碼撰寫任務，不涉及需驗證的外部知識
- 已查過且結論明確的重複問題，避免重複消耗資源
- 單純尋求個人意見或主觀建議，沒有客觀文件可查

## 執行流程

1. **範圍與複雜度評估**：由 main agent 內部執行，輸出限制在 3 至 5 行條列，避免冗長自我對話。需釐清：使用者要解決的核心問題、需要查找哪些面向才能完整回答、預估每個面向的材料規模。
1. **分流決定**：依據評估結果判定使用單一 researcher 模式或 fan-out 模式。判定原則詳見「分流決定」章節。
1. **執行搜尋**：委派一個或多個一手研究 subagent 並行執行，每個 subagent 收到一個明確的子面向 prompt。
1. **整合判定**：依據 subagent 數量與材料量判定是否啟動二手整合 subagent。判定原則詳見「整合判定」章節。
1. **結果交付**：Main agent 從最終的暫存檔讀取整合結果，組織回應後交付使用者。
1. **暫存檔清理**：任務完成後 main agent 主動清除所有 `.tmp` 暫存檔，依 `tmp-file-usage` rule 規定。

## 拆解原則

Fan-out 模式拆分子面向時，必須遵守以下規範：

- **互斥**：兩個面向不應有大量重疊，避免多個 subagent 重複搜尋同一份權威來源。
- **可獨立完成**：每個面向應能不依賴其他面向就能查清，避免 subagent 之間需互通有無。
- **粒度合宜**：每個面向應對應一個能獨立寫成段落的子主題，不應細到關鍵字級別。
- **數量上限**：面向數超過 4 個時應重新評估是否拆得太細，這是軟性提醒，不是硬限制。

## 分流決定

- **單一 researcher 模式**：問題本質為單一明確主題，且材料量預期不大，使用單一一手研究 subagent。
- **Fan-out 模式**：問題涉及多個彼此獨立的面向，或需要從多個來源做交叉驗證，依對應拆解出的面向數量啟動多個一手研究 subagent 並行執行。

## 整合判定

- **多個 subagent**：一律啟動二手整合 subagent 做跨來源彙整。
- **單一 subagent 且材料量小**：粗略估計 1000 token 以內，由 main agent 直接讀取暫存檔，不啟動整合 subagent。
- **單一 subagent 但材料量大或內容雜亂**：仍啟動二手整合 subagent 做精煉，避免 main agent 直接處理大量原始材料。

## 委派 Prompt 要求

傳給一手研究 subagent 的 prompt 必須包含：

- 此 subagent 負責的明確子面向描述
- 搜尋目標，說明需要找到什麼層級的資料
- 已知約束，例如指定的技術版本或必須涵蓋的特定 API

傳給二手整合 subagent 的 prompt 必須包含：

- 整合目標的明確描述
- 所有相關研究材料的暫存檔路徑清單
- 預期輸出格式重點
- 任何已知的優先級資訊，例如哪個版本是必須涵蓋的

兩類 prompt 均不得指定 subagent 內部該用哪個工具或哪條搜尋詞，由 subagent 自行決定。

## 材料不足與補搜尋

二手整合 subagent 在整合時若發現某面向材料不足或有未解的衝突，應在報告中明確標示「此處材料不足，建議補搜尋」並描述缺口內容。是否補搜尋由 main agent 判斷，必要時 main agent 啟動新一輪一手研究 subagent 處理特定缺口。

本 skill 不支援整合 subagent 自動觸發補搜尋，以避免形成失控迴圈。

## 輸出格式

Main agent 最終回應使用者時應包含：

- 整合後的研究結論，依主題組織
- 來源歸屬摘要，說明主要結論來自哪些來源
- 若有未解的不確定性或材料缺口，明確標示
