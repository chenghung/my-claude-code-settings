## 設計原則

此節列出 LLM 產出 agents、rules、skills、commands、prompt 相關檔案時必須遵守的設計原則，共八條，依優先序由高到低排列。編號越小優先順序越高。當原則之間產生取捨時，優先滿足編號較小者。

### 具體明確、無規則衝突

prompt 指令必須具體明確，使 LLM 能百分之百依據指令完成任務，並禁止與其他 rule、agent、skill 之間出現互相矛盾的行為要求。

### 精簡、避免重複

在滿足第一條的前提下，盡可能減少重複或不必要的 prompt 內容。重複的指令會浪費 token，且可能在不同上下文中產生不一致的行為。

### 動態載入、避免污染 main agent context

動態載入額外資訊到 context window 時，必須避免污染 main agent 的主要目的。與主要目的非高度相關的內容，應盡量透過 skill 或 subagent 的方式動態載入，而不是常駐在 main agent 的 context 中。能委派給 subagent 處理的工作就盡可能委派。

### 單一職責、觸發條件明確

每個 agent、rule、skill 只承擔單一明確職責，其 description 或觸發條件必須讓 LLM 能無歧義地判斷何時該呼叫、何時不該呼叫。職責重疊是規則衝突的主要來源，屬於第一條能否成立的前置條件。

### Subagent 互相獨立、不交叉引用

每個 subagent 的 prompt 必須假設自己是獨立存在的，不得在內文中引用、提及或建議使用其他 subagent 的名稱。Subagent 被呼叫時是無狀態且獨立的，不知道其他 subagent 是否存在於當前環境，引用其他 subagent 等於做了一個它無法驗證的假設。職責邊界的說明應以「不處理什麼類型的問題」描述，而非「建議改用某某 subagent」。路由與分流是 main agent 或 skill 層級的責任，不是 subagent 的責任。

### 可驗證的成功條件

prompt 必須定義任務完成的判斷標準與輸出格式，例如應產出哪些檔案、應回傳哪些欄位、遇到哪些狀況應停下並詢問使用者。缺乏成功條件會讓 LLM 憑感覺收尾，main agent 也無法驗證 subagent 是否確實完成任務。

### 邊界與失敗行為明確

prompt 必須涵蓋越界輸入、資訊不足、工具呼叫失敗等情境下的應對方式，例如詢問使用者、回報失敗、或採用預設值。未定義的邊界行為會導致 LLM 自行腦補，產生跨 session 不一致的結果。

### 委派時只給意圖，不指揮內部實作

main agent 委派給 subagent 時，只描述目標與預期產出，不指定 subagent 內部應使用哪些工具或執行哪些步驟。抽象層級一致才能讓 subagent 發揮專長，也避免 main agent 的假設污染 subagent 的決策空間。

## 觸發條件

每當修改了以下任何一項設定後，必須立即執行本規則所定義的完整檢視流程：

- `skills/` 目錄下的任何檔案
- `agents/` 目錄下的任何檔案
- `rules/` 目錄下的任何檔案
- `.claude/rules/` 目錄下的任何檔案
- `CLAUDE.md`

## 檢視項目

### 上下文規則衝突檢查

確認新修改的內容與現有的其他 rules、agent 定義、skill 定義之間，沒有互相矛盾的指令或行為要求。任何衝突都必須在完成本次修改前解決。

### Subagent 交叉引用檢查

確認 `agents/` 目錄下任何 subagent 定義檔案的內文中，沒有出現其他 subagent 的名稱或對其他 subagent 的引用。職責邊界的描述必須以問題類型表達，而非以其他 subagent 名稱表達。發現此類引用必須立即移除或改寫。

### 幻覺風險檢查

確認所有 prompt 中沒有引用不存在的工具、不存在的檔案路徑、不存在的功能，或描述與實際行為不符的內容。發現任何此類問題，必須立即更正。

### 重複 prompt 檢查

找出跨檔案之間是否存在重複或高度相似的 prompt 片段。重複的指令會浪費 token，且可能在不同上下文中產生不一致的行為，必須合併或刪除冗餘內容。

### 安全憑證檢查

嚴格禁止將任何敏感的安全憑證直接置於 `skills/`、`agents/`、`rules/` 或 `.claude/rules/` 目錄下的任何檔案中。

禁止放入的內容包含但不限於：API 金鑰、密碼、存取權杖（access token）、含有 token 的 webhook URL、私密金鑰（private key）、任何形式的 secret。

此類資訊必須透過環境變數或外部的 secrets 管理機制來處理，不得以明文形式寫入版本控制的檔案中。

若在檢視過程中發現既有檔案包含上述內容，必須立即向使用者提出警告，並建議以安全的替代方案取代。

## 檢視範圍

執行檢視時，必須涵蓋以下所有相關檔案：

- `rules/` 目錄中的全部 `.md` 檔案
- `.claude/rules/` 目錄中的全部 `.md` 檔案
- `agents/` 目錄中的全部 agent 定義檔案
- `skills/` 目錄中的全部 skill 定義檔案
- 根目錄與 `.claude/` 目錄下的 `CLAUDE.md`

## 執行要求

若在上述任何檢視項目中發現問題，必須在完成本次修改前主動提出並修正，不得將問題遺留至後續作業。
